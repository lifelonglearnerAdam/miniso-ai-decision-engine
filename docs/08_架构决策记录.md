# 架构决策记录（ADR 摘要）

本页面向企业技术与治理评审人员，说明竞赛验证版的关键技术取舍、理由、可验证后果及生产化前仍需补充的控制。

## ADR-001：把模型与评估协议解耦

- 状态：接受
- 决策：Agent 通过统一客户端调用模型；回测、校准和多目标搜索可以在无 LLM 时独立运行。
- 原因：基座模型可替换，真正需要版本化的是数据快照、Prompt 契约、评估协议和决策记录。
- 后果：演示可离线复现；上线时必须补模型网关和统一身份。

## ADR-002：候选预测前移除结果列

- 状态：接受
- 决策：回测引擎在调用预测器前删除 `is_hit`、`sales_90d`、`realized_margin`。
- 原因：仅靠建模者自律无法形成防泄漏控制。
- 后果：预测器接口更安全；企业接入时仍需字段级可用时间和派生特征审计。

## ADR-003：滚动固定窗而非扩展窗

- 状态：接受
- 决策：默认 180 天固定训练窗、14 天隔离、90 天预测、30 天步长。
- 原因：固定窗能更清楚观察概念漂移，隔离窗模拟数据迟到和研发决策空白期。
- 后果：参数必须按品类周期重新验证，不能把默认值当行业真理。

## ADR-004：虚拟面板只作为待校准弱信号

- 状态：接受
- 决策：用独立留出残差构造保形区间，禁止把原始 LLM 人设评分直接解释为购买概率。
- 原因：LLM 人群模拟存在趋同、谄媚和分布失真风险。
- 后果：没有真实锚点时，系统只能输出定性洞察，不能给“已校准概率”。

## ADR-005：高影响动作必须人工批准

- 状态：接受
- 决策：系统只生成候选和证据包；采购、定价、打样、上架与外部发布均在自动化边界之外。
- 原因：这是产品研发决策支持系统，不是自治经营主体。
- 后果：审批体验和证据可解释性是产品能力，不是额外负担。
